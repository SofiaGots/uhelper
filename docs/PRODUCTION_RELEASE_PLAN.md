# Production Release Plan for UHelper (Telegram MVP → Web)

## Контекст
- Цель: стабильный Telegram-бот к февралю 2026 с базой университетов и персональными рекомендациями; дальнейшая миграция на веб после выпуска.
- Технологии по договоренностям: Python + Aiogram + LangChain/LangGraph, строгая типизация, Pydantic v2, async I/O, `uv` + `ruff` (black-стиль), тесты `pytest`, проверки `ty`.
- Главные компоненты: OrchestratorAgent, UniversityDataAgent, ProfileAnalyzerAgent, TimelineAgent (MVP), NotificationAgent (минимально для напоминаний).

## Критические артефакты
- `main.py` — точка входа.
- `src/bot.py` — телеграм-обвязка.
- `src/agents/` — логика агентов; приоритет: orchestrator, university_data, profile_analyzer, timeline.
- `src/models/` — Pydantic v2 модели (без сырых dict).
- `pyproject.toml`, `uv.lock` — зависимости и инструменты (`ruff`, `pytest`, `ty`).
- `.env.example` — переменные окружения; все параметры должны валидироваться моделью конфигурации.

## Неделя 1: Базовая гигиена и тесты
- [ ] Создать `src/config.py` с Pydantic `Settings`: токен бота, AI ключ, БД URL, Redis URL (опц.), режимы dev/prod; добавить `.env.example`.
- [ ] Подключить `ruff` и `ty` в `pyproject.toml`; команды `uv run ruff check`, `uv run ruff format`, `uv run ty`; зафиксировать в README.
- [ ] Привести `src/agents/*.py`, `src/bot.py` к полной типизации и async I/O; убрать sync-файловые и сетевые вызовы.
- [ ] Добавить базовый `logging` конфиг (stdout, уровни INFO/ERROR; формат с request_id/session_id).
- [ ] Написать юнит-тесты: OrchestratorAgent маршрутизация намерений по заранее размеченному набору; UniversityDataAgent фильтры по городу/направлению; ProfileAnalyzerAgent расчёт GPA/сил предметов.
- [ ] Создать общие фикстуры `pytest`: фейковый `ai_client`, фейковый telegram update, временная файловая БД/JSON.
- [ ] Включить `pytest.ini` с опциями async (`pytest-asyncio`), ограничить длительность тестов (`--maxfail=1`).

## Неделя 2: Надёжность и безопасность для MVP
- [ ] Добавить санитайз входного текста (обрезка до лимита, фильтр html/escape, язык ru/en) и валидацию в моделях запросов.
- [ ] Реализовать middleware rate limiting (in-memory dict; интерфейс под Redis) + тесты на превышение лимита.
- [ ] Добавить ретраи и таймауты для AI/API вызовов; простой circuit breaker (счётчик ошибок + backoff).
- [ ] Унифицированные ошибки: свои исключения для агентов; пользователь получает безопасное сообщение, лог — стек и контекст.
- [ ] Маскирование токенов в логах; структура логов: уровни, session_id, intent, агент.
- [ ] Healthcheck handler (возвращает версию и статус зависимостей); e2e тест на него.

## Неделя 3: Данные и инфраструктура запуска
- [ ] Подготовить минимальный датасет вузов (CSV/JSON): название, город, направления, дедлайны, стипендии; Pydantic-схема `University`.
- [ ] Репозиторий данных: класс для загрузки/кэша в память; тесты на фильтрацию и валидацию дат.
- [ ] Скрипт и команда `uv run python -m scripts.load_universities` для первичного наполнения (и пример файла).
- [ ] Dockerfile (python slim, uv, non-root user); docker-compose с опциями Redis (по флагу) и volume для данных.
- [ ] GitHub Actions workflow: шаги lint → типы → тесты → сборка Docker; артефакт образа.
- [ ] Выбрать и зафиксировать метод получения апдейтов: polling (MVP) или webhook; реализовать healthcheck endpoint.

## Неделя 4: Пользовательский опыт и аналитика
- [ ] Сценарии: команда поиска вуза по интересам; персональная рекомендация на основе профиля; создание таймлайна по выбранным вузам.
- [ ] Улучшить формулировки ответов агентов (без упоминаний AI; нейтральный стиль, кратко).
- [ ] Собор метрик: время ответа (p50/p95), точность намерений (по тест-набору), количество ошибок агентов; запись в файл + stdout.
- [ ] Команда `/feedback` → сохраняет текст и метаданные в файл/БД; тесты на корректность записи.
- [ ] Простые алерты: логирование WARN при p95 > порога или росте ошибок AI вызовов; уведомление в консоль/лог.

## Неделя 5: Документация и подготовка к бете
- [ ] Добавить краткие Google-style docstrings к публичным методам агентов и моделей.
- [ ] Обновить README: запуск через `uv`, генерация `.env`, обновление датасета, сценарии команд.
- [ ] Написать инструкцию по сборке/деплою Docker-образа (команды, переменные окружения, обновление токенов).
- [ ] Сформировать чек-лист перед выкладкой: `ruff`, `ty`, `pytest`, обновление данных, проверка токенов, миграции (если есть).
- [ ] Подготовить шаблон release notes для беты (изменения, известные ограничения, как дать фидбэк).

## Неделя 6: Бетатест и стабилизация
- [ ] Нагрузочный тест (эмуляция 200 пользователей, 10 мин): измерить p50/p95, ошибки, потребление CPU/RAM.
- [ ] Финальные e2e сценарии: поиск вузов, персональная рекомендация, напоминание по срокам; фиксация чек-листа прохождения.
- [ ] План отката: хранить предыдущий Docker образ + резервная копия данных; документировать шаги возврата.
- [ ] Сбор фидбэка пилотов: форма/бот-команда; triage багов по приоритетам P0/P1/P2; быстрые фиксы в патч-релизе.

## Критический чек-лист «выпускаем в прод»
- [ ] Все операции I/O — async; нет блокирующих вызовов в хэндлерах Telegram.
- [ ] Только Pydantic v2 модели, строгая валидация входов и env.
- [ ] Линт/типы/тесты зелёные (`uv run ruff check`, `uv run ty`, `uv run pytest`).
- [ ] Секреты только из env; `.env` не в git; токены ротуются перед релизом.
- [ ] Логи без персональных данных; токены маскируются.
- [ ] Rate limiting включён; healthcheck работает; базовое мониторинг-логирование в stdout/файл.
- [ ] Docker-образ собирается и стартует; есть пошаговая инструкция деплоя и отката.

## Метрики успеха (MVP)
- Ответ бота < 2 c в 95-м перцентиле; точность маршрутизации намерений > 90% на тестовом наборе.
- Не менее 80% корректных рекомендаций по университетам на ручной проверке пилотами.
- Ошибки 5xx < 1% запросов; отказоустойчивость при падении внешнего AI (graceful degradation).

## Риски и смягчение
- Данные университетов устаревают → сценарий обновления датасета + дата последнего обновления в логах.
- Латентность AI → кэширование ответов по ключевым запросам, таймауты и повтор.
- Утечка токенов → .env в gitignore, токены только через переменные окружения, ротация перед бетой.
- Ограничения Telegram API → rate limit, очереди сообщений и деградация при превышении.

## Примечания
- Используем только разрешённые инструменты; новые библиотеки — через PR и обсуждение.
- Один коммит — одна логическая задача; все изменения через feature-ветки и PR в `main` запрещены напрямую.
- Минимизируем комментарии; сложные места снабжаем краткими пояснениями.
